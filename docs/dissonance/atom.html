RNNoise is a noise suppression system, developed by Jean-Marc Valin, which uses machine learning to learn separate useful signal (human speech) from noise (e.g. barking dogs). RNNoise splits the audio up into 22 frequency bands, decides how noisy each band is, and then attenuates the band based on this. This process repeats again every 10 milliseconds, adjusting the volume of each frequency band to minimise noise. This defines exactly what the three numbers in the version mean:</p> <pre><code>Version `MAJOR.MINOR.PATCH` - Increment MAJOR version when you make incompatible API changes. - Increment MINOR version when you add functionality in a backwards-compatible manner. - Increment PATCH version when you make backwards-compatible bug fixes. </code></pre><p>A <code>MAJOR</code> change means that a change to Dissonance might <em>break</em> your code which interfaces with Dissonance. Naturally we try to keep such changes to a minimum. When they do happen we make sure to draw attention to it in the release notes and explain how best to handle it.</p> <p>With <a href="/dissonance/releases/6.1.0.html">Dissonance 6.1.0</a> we included what we considered a very minor bugfix:</p> <blockquote> <p>Always using network order (Big endian) for network packet headers</p> </blockquote> <p>One number in the network protocol used &quot;little endian&quot; encoding instead of the more standard &quot;big endian&quot; encoding used on networks. We didn&#39;t consider this a <code>MAJOR</code> or even a <code>MINOR</code> change because the network protocol is an implementation detail; your code never reads or writes packets so therefore a change to the protocol can&#39;t possibly break your code.</p> <p>However, after talking to users about the consequences of this tiny change we&#39;ve realised that this wasn&#39;t the correct way to think about it. Although a change to the network protocol does not break any <em>code</em> it can lead to a very difficult deployment which forces end users of your application to either upgrade or be unable to communicate.</p> <p>We have decided that in the future we will include changes to the network protocol in the version number - i.e. an incompatible change to the network will result in the <code>MAJOR</code> version number increasing. Furthermore since an incompatible change to the network protocol is very difficult for you to deploy to your users we consider a network protocol change to be one of the worst kind of changes we can make.</p> <p>From this point forwards <strong>we don&#39;t anticipate <em>ever</em> needing to make an incompatible change to the protocol</strong>.
